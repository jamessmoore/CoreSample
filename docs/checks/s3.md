# S3 audit checks

Implemented in `s3-audit-mcp/audit.py` (`S3Auditor`). Four checks, each its
own key in the JSON returned by the `audit_s3` tool. Unlike IAM, S3 buckets
are regional resources -- each bucket's actual region (resolved via
`get_bucket_location`) is matched against the `region` parameter before any
check runs, the same way `audit_ec2` only audits that region's instances.

## Public bucket access

**What it checks** — Buckets reachable via a public ACL grant (`AllUsers`
or `AuthenticatedUsers` group URI on `get_bucket_acl`) or a public bucket
policy (`get_bucket_policy_status`'s `IsPublic`).

**Why it matters** — A public bucket is directly readable (and sometimes
writable) by anyone on the internet. This is the single most common cause
of real-world S3 data breaches.

**Severity** — `critical`

**Example finding**

```json
{
  "bucket_name": "example-bucket",
  "severity": "critical",
  "recommendation": "Remove public ACL grants and/or the public bucket policy immediately"
}
```

**Remediation** — Remove the public ACL grant and/or the offending bucket
policy statement. Enable S3 Block Public Access (see the next check) as a
backstop so this can't silently recur.

## Public access block not fully enabled

**What it checks** — Buckets where any of the four S3 Block Public Access
settings (`BlockPublicAcls`, `IgnorePublicAcls`, `BlockPublicPolicy`,
`RestrictPublicBuckets`) is disabled or not configured at all
(`get_public_access_block`).

**Why it matters** — Block Public Access is the safety net that prevents a
future ACL or policy change from accidentally making the bucket public.
Without all four settings enabled, that safety net has gaps.

**Severity** — `high`

**Example finding**

```json
{
  "bucket_name": "example-bucket",
  "missing_protections": ["RestrictPublicBuckets"],
  "severity": "high",
  "recommendation": "Enable all four S3 Block Public Access settings for this bucket"
}
```

**Remediation** — Enable all four Block Public Access settings at the
bucket level (or account level, which covers every bucket at once).

## Default encryption not enabled

**What it checks** — Buckets with no default server-side encryption
configuration (`get_bucket_encryption` returns no rules, or raises
`ServerSideEncryptionConfigurationNotFoundError`).

**Why it matters** — Without default encryption, objects uploaded without
an explicit encryption header are stored in plaintext. Default encryption
closes that gap for every future upload regardless of how it's written.

**Severity** — `medium`

**Example finding**

```json
{
  "bucket_name": "example-bucket",
  "severity": "medium",
  "recommendation": "Enable default server-side encryption (SSE-S3 or SSE-KMS) on this bucket"
}
```

**Remediation** — Enable default encryption (SSE-S3 is sufficient for most
cases; use SSE-KMS if key-level access control or audit logging via
CloudTrail data events is required).

## Versioning not enabled

**What it checks** — Buckets where versioning isn't `Enabled`
(`get_bucket_versioning`'s `Status` is absent or `Suspended`).

**Why it matters** — Without versioning, an accidental delete or overwrite
is unrecoverable. Versioning is the cheapest available protection against
both accidental data loss and a subset of ransomware-style attacks that
overwrite objects in place.

**Severity** — `medium`

**Example finding**

```json
{
  "bucket_name": "example-bucket",
  "severity": "medium",
  "recommendation": "Enable versioning to protect against accidental deletion or overwrite"
}
```

**Remediation** — Enable versioning on the bucket. Pair with a lifecycle
rule to expire old versions after a retention window if storage cost is a
concern.
